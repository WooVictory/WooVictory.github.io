ActivityLifeCycle

Activity

애플리케이션의 가장 기초적인 구성요소로서 스마트폰 사용자가 일을 하기 위해 상호작용할 수 있는 화면을 제공한다. 즉, 뷰를 그릴 수 있는 캔버스

하나의 Application은 여러 개의 Activity가 느슨하게 묶여져 있습니다. Activity는 여러가지 작업을 수행하기 위해서 다른 Activity를 시작할 수 있고, 새로운 Activity가 시작할 때마다 이전 Activity는 중단(stop)이 된다. 시스템은 중단된 Activity를 BackStack에 보존하고 있습니다. 그리고 새로운 Activity가 종료될 경우(Destroy) BackStack에 있던 Activity가 다시 로딩되서 화면에 보이게 됩니다(Restart). 이런 일련의 과정을 Activity 생명주기라고 하며 아래의 링크로 가서 내용을 확인할 수 있습니다.

안드로이드 생명주기 : https://woovictory.github.io/2018/03/02/20180302Android1/

왜 생명주기가 필요할까??

  1. 사용자가 앱을 사용하는 도중에 전화가 걸려오거나 다른 앱으로 전환할 때 충돌하지 않도록 하기 위함입니다.
  2. 사용자가 앱을 활발하게 사용하지 않는 경우, 소중한 시스템 리소스를 소비하지 않도록 합니다.
  3. 사용자가 앱에서 나갔다가 나중에 돌아왔을 때 사용자의 진행 상태를 손실하지 않도록 합니다.
  4. 화면이 가로 방향과 세로 방향 간에 회전할 경우, 충돌하거나 사용자의 진행 상태를 손실하지 않도록 합니다.

복습을 하기 위해서 강의를 들으면서 다시 정리해보겠습니다.

  • onCreate() : Activity가 생성될 때 한번만 호출됩니다.
  • onStart() : onCreate()가 끝난 다음에 호출되는 함수이고, View가 그려지기 전에 호출되는 함수이다.
  • onResume() : View가 그려지고 나서 호출된다. 사용자와 Activity가 통신하는 상태로서 앱이 활동하는 단계라고 생각할 수 있습니다.
  • onPause() : Activity가 반만 가려졌을 경우 호출되는 함수입니다.
  • onStop() : Activity가 완전히 닫힌 경우나 완전히 가려진 경우 호출되는 함수입니다
  • onDestroy() : 안드로이드 OS나 JVM이 메모리를 정리할 때 자동적으로 호출하는 함수입니다.

여기서 Activity LifeCycle이 중요한 이유는 전화를 받았다가 끄고 다시 원래 화면으로 돌아올 경우나 다른 Activity에서 원래 화면으로 돌아올 경우 onCreate() 부터 시작하지 않고 onResater() -> onStart() -> onResum()을 거쳐서 함수가 실행되기 때문에 빠르게 돌아올 수 있고, 이 순환이 반복됩니다.

Intent

일종의 메시지 객체로서 이를 통해서 다른 앱의 구성요소나 액티비티 작업을 요청할 수 있습니다.
기본적으로는 아래와 같은 3가지 방식으로 사용됩니다.

  1. Activity 시작
  2. 서비스 시작
  3. 브로드캐스트 전달

Activyt는 App안의 단일 화면을 나타냅니다. 만약, A Activity에서 B Activity로 넘어가기 위해서는 Intent에 B의 정보를 담고 startActivity() 함수를 호출하면 Android System은 새로운 B Activity를 만들게 됩니다.

Permission

승인, 권한이라는 의미이며 안드로이드 OS에서 애플리케이션 권한을 획득해야 사용자 디바이스에 설치된 기능을 사용할 수 있습니다. Permission은 Manifest.xml 파일에 등록합니다.
아래는 자주 사용하는 Permission의 종류에 대한 정리입니다.
다양한 Permission이 궁금하다면 Google에 Android Permission을 검색하면 많은 권한을 확인할 수 있습니다.

Request Permissions

안드로이드 6.0(API 23 이상)부터는 사용자는 App이 설치될 때 권한을 부여하는 것이 아니고 App이 실행될 때 권한을 부여하게 됩니다. 이유는 App이 설치될 때 권한이 부여될 경우, 사용자의 기밀 데이터가 유출될 염려가 있기 때문입니다. Application 실행 중에 위험 권한을 넘겨줘도 되는지 사용자에게 요청하기 위함입니다.

Requset Permission 즉, 위험 권한의 종류는 개인 정보와 관련된 권한이며 총 9가지가 있습니다. API 23 즉 마시멜로우 이상부터는 사용자의 허가를 받아야 정보에 접근할 수 있습니다.

  1. 달력 : 사용자의 스케줄이 유출 될 수 있습니다.
  2. 카메라 : 사진을 찍을 때 뿐만 아니라 사진에 접근할 때 유출될 가능성
  3. 전화 : 전화 수신, 또는 통화목록 조회, 전화번호 접근
  4. 마이크 : 마이크 권한을 잘못 사용할 경우 도청
  5. 위치정보 : 위치 정보를 잘못 사용하면 위험
  6. SMS : 문자 내역 유출
  7. Storage : 중요 정보 유출
  8. Sensor : 사용자의 신체 관련된 값을 가져올 때 사용
  9. 연락처 : 개인 정보 유출
