ActivityLifeCycle

Activity

애플리케이션의 가장 기초적인 구성요소로서 스마트폰 사용자가 일을 하기 위해 상호작용할 수 있는 화면을 제공합니다. 즉, 뷰를 그릴 수 있는 캔버스입니다.

하나의 Application은 여러 개의 Activity가 느슨하게 묶여져 있습니다. Activity는 여러가지 작업을 수행하기 위해서 다른 Activity를 시작할 수 있고, 새로운 Activity가 시작할 때마다 이전 Activity는 중단(stop)이 됩니다. 시스템은 중단된 Activity를 Back Stack에 보존하고 있습니다. 그리고 새로운 Activity가 종료될 경우(Destroy) BackStack에 있던 Activity가 다시 로딩되서 화면에 보이게 됩니다(Restart). 이런 일련의 과정을 Activity 생명주기라고 하며 아래의 링크로 가서 정리한 내용을 확인할 수 있습니다.

안드로이드 생명주기 : https://woovictory.github.io/2018/03/02/20180302Android1/

왜 생명주기가 필요할까??

  1. 사용자가 앱을 사용하는 도중에 전화가 걸려오거나 다른 앱으로 전환할 때 충돌하지 않도록 하기 위함입니다.
  2. 사용자가 앱을 활발하게 사용하지 않는 경우, 소중한 시스템 리소스를 소비하지 않도록 합니다.
  3. 사용자가 앱에서 나갔다가 나중에 돌아왔을 때 사용자의 진행 상태를 손실하지 않도록 합니다.
  4. 화면이 가로 방향과 세로 방향 간에 회전할 경우, 충돌하거나 사용자의 진행 상태를 손실하지 않도록 합니다.

복습을 하기 위해서 강의를 들으면서 간단하게나마 다시 정리해보았습니다.

  • onCreate() : Activity가 생성될 때 한번만 호출됩니다.
  • onStart() : onCreate()가 끝난 다음에 호출되는 함수이고, View가 그려지기 전에 호출되는 함수이다.
  • onResume() : View가 그려지고 나서 호출된다. 사용자와 Activity가 통신하는 상태로서 앱이 활동하는 단계라고 생각할 수 있습니다.
  • onPause() : Activity가 반만 가려졌을 경우 호출되는 함수입니다.
  • onStop() : Activity가 완전히 닫힌 경우나 완전히 가려진 경우 호출되는 함수입니다
  • onDestroy() : 안드로이드 OS나 JVM이 메모리를 정리할 때 자동적으로 호출하는 함수입니다.

여기서 Activity LifeCycle이 중요한 이유는 전화를 받았다가 끄고 다시 원래 화면으로 돌아올 경우나 다른 Activity에서 원래 화면으로 돌아올 경우 onCreate() 부터 시작하지 않고 onResater() -> onStart() -> onResum()을 거쳐서 함수가 실행되기 때문에 빠르게 돌아올 수 있고, 이 순환이 반복됩니다.

Intent

일종의 메시지 객체로서 이를 통해서 다른 앱의 구성요소나 액티비티 작업을 요청할 수 있습니다.
기본적으로는 아래와 같은 3가지 방식으로 사용됩니다.

  1. Activity 시작
  2. 서비스 시작
  3. 브로드캐스트 전달

Activity는 App안의 단일 화면을 나타냅니다. 만약, A Activity에서 B Activity로 넘어가기 위해서는 Intent에 B의 정보를 담고 startActivity() 함수를 호출하면 Android System은 새로운 B Activity를 만들게 됩니다.

Permission

'승인, 권한’이라는 의미이며 안드로이드 OS에서 애플리케이션 권한을 획득해야 사용자 device에 설치된 기능을 사용할 수 있습니다. Permission은 Manifest.xml 파일에 등록합니다.
아래는 자주 사용하는 Permission의 종류에 대한 정리입니다.

다양한 Permission이 궁금하다면 Google에 Android Permission을 검색하면 많은 권한을 확인할 수 있습니다.

Request Permissions

안드로이드 6.0(API 23 이상)부터는 사용자는 App이 설치될 때 권한을 부여하는 것이 아니고 App이 실행될 때 권한을 부여하게 됩니다. 이유는 App이 설치될 때 권한이 부여될 경우, 사용자의 기밀 데이터가 유출될 염려가 있기 때문입니다. Application 실행 중에 위험 권한을 넘겨줘도 되는지 사용자에게 요청하기 위함입니다.

Permission 즉, 위험 권한의 종류는 개인 정보와 관련된 권한이며 총 9가지가 있습니다. API 23 즉 마시멜로우 이상부터는 사용자의 허가를 받아야 정보에 접근할 수 있습니다.

  1. 달력 : 사용자의 스케줄이 유출 될 수 있습니다.
  2. 카메라 : 사진을 찍을 때 뿐만 아니라 사진에 접근할 때 유출될 가능성
  3. 전화 : 전화 수신, 또는 통화목록 조회, 전화번호 접근
  4. 마이크 : 마이크 권한을 잘못 사용할 경우 도청
  5. 위치정보 : 위치 정보를 잘못 사용하면 위험
  6. SMS : 문자 내역 유출
  7. Storage : 중요 정보 유출
  8. Sensor : 사용자의 신체 관련된 값을 가져올 때 사용
  9. 연락처 : 개인 정보 유출

Fragment

Fragment는 간단하게 말해서 추가 및 제거가 가능한 모듈식 액티비티 Section이라고 볼 수 있습니다. 즉, Activity에서 부분 화면을 나타내기 위해서 사용하는 것입니다. 자체 생명 주기를 가지고 자체 입력 이벤트를 가지고 독립적으로 활동할 수 있지만, Context는 존재하지 않기 때문에 Activity로부터 받아와야 합니다.

  • 생명주기
    Fragment를 관리하는 것은 Activity를 관리하는 것과 비슷합니다.
    Activity와 마찬가지로 Fragment는 6가지의 상태가 있습니다.

  1. 상태
    Created
    Started
    Resumed
    Paused
    Stopped
    Destroyed

  2. 함수
    onCreateView() : View가 만들어질 때 호출됩니다
    onStartView() : 가 화면에 그려지기 직전에 호출
    onResume() : 그려지고 나서 호출된다.
    onPause() : 반 투명 상태나 화면이 반만 가려졌을 때 호출
    onStop() : 화면에서 Fragment가 없어졌을 때 호출, Fragment가 사라져도 BackStack에 남아있을 때 호출되는 부분
    onDetach() : Fragment가 Frame에서 완전히 제거되었을 때 Fragment도 사라지고 BackStack에서도 사라졌을 때 호출되는 부분

Fragment 생명주기는 11가지가 있지만, 여기서는 자주 사용하는 6가지만 사용해보도록 하겠습니다.

Context

안드로이드에서 가장 많이 쓰이는 객체이며, Context의 개념은 모호하고 어렵고 무엇이라고 정의하기는 어렵습니다. 본래 뜻은 문맥, 전후 관계, 배경, 상황입니다. 즉, 이 화면에 대한 전체적인 상황을 가지고 있는 객체라고 생각할 수 있습니다.

Context를 다른 말로는 현재 사용되고 있는 애플리케이션(또는 액티비티)에 대한 포괄적인 정보를 지니고 있는 객체라고 할 수 있습니다.
this는 자기자신의 클래스를 나타내는 것이고 이 클래스 안에 있는 AppCompatActivity를 가리켜서 이 토스트 박스를 이 액티비티에 띄우겠다는 명려어입니다.

SharedPreference

안드로이드에서 App을 사용하다가 종료하게 되면 보통 실행된 데이터가 초기화됩니다. 하지만, 예를 들어서 장바구니, 로그인 아이디, AccessToken과 같은 재사용해야 하는 데이터를 저장해야 될 경우에 SharedPreference에 저장해서 사용하게 됩니다.

하지만, 안드로이드는 SQLite가 존재하지만 복잡한 스키마 구조나 대용량 데이터를 저장할 경우에만 사용하고 나머지는 SharedPreference로 데이터를 저장합니다. 데이터는 App이 삭제되기 전까지 보존됩니다.
Key-Value 형태로 데이터를 저장하게 됩니다. 이를 읽거나 쓸 수 있는 메소드를 제공하고, 파일은 XML 방식으로 저장됩니다.

Handler

Thread는 간단하게 말하면 여러 작업을 동시에 수행할 수 있도록 도와줍니다.

위의 그림과 같이 뉴스, 현재위치, 실시간 검색어를 가져오는 기능을 가진 앱을 만든다고 가정해보았습니다.
Thread가 없을 경우에는, 뉴스를 가져온 다음에 현재 위치를 가져온 다음에, 실시간 검색어를 가져오고 나서야 앱이 실행됩니다. 각각 수행시간이 10초, 10초, 5초가 걸린다면 부팅 시간이 총 25초가 걸리면서 사용자에게 불편을 초래할 수 있고, App의 반응 속도에도 좋지 않은 영향을 주게 됩니다.

이와 같은 문제를 해결하기 위해서 뉴스를 가져오면서 현재 위치를 가져옥 동시에 실시간 검색어까지 가져올 수 있도록 도와주는 것을 Thread라고 하며 이 Thread가 동시에 작동하는 것을 멀티 스레딩이라고 합니다.

굉장히 자주 쓰이고 유용한 기능이지만 문제를 발생시킵니다. 각각의 뉴스, 현재 위치, 실시간 검색어의 내용이 텍스트에 표시된다고 구현했다고 가정을 했을 때, 뉴스와 현재 위치의 내용이 동시에 끝났을 경우 텍스트는 동시에 두가지 내용을 받게 됩니다. 그러면 텍스트는 두 가지 내용 중에 어떤 것을 받을지 갈피를 잡지 못하고 에러를 발생시키고 앱이 종료됩니다. 이러한 Thread의 충돌을 해결해주는 것이 Handler입니다.

Thread는 Handler 없이 Main Thread를 건드릴 수 없기 때문에 에러가 발생합니다. 그래서 꼭 Handler를 사용해줘야 합니다.

AsyncTask

Async란 비동기라는 뜻으로 어떠한 작업이 순차적으로 진행되는 것이 아니라 동시에 진행될 때 Async라는 말을 사용합니다. 즉, Thread + Handler = AsyncTask 라고 이해하면 쉽습니다.

FCM

푸쉬알람(FCM)은 카카오톡을 사용할 때 메시지가 도착하는 알림 메시지와 같은 것입니다.

FCM의 장점

  • 비용이 따로 발생하지 않습니다.
  • FCM을 사용하는 단말기의 배터리 절약이 가능합니다.
  • 다른 플랫폼과 호환이 가능합니다. FCM으로 iOS 단말기나 Chrome 브라우저에 푸쉬를 보낼 수 있습니다.

메시지를 날리는 서버를 구축했을 경우 메시지는 바로 단말기로 전송되는 것이 아니라 메시지는 Google Cloud Server를 거쳐서 전송되게 됩니다.

이러한 복잡한 매커니즘을 사용하는 이유는??

만약 단말기에 여러 종류의 App이 설치 되어 있고, 직접적으로 푸쉬 알람을 날릴 경우 단말기는 많은 서버와 통신을 해야 되는 부담감을 가지고 있습니다. 이것은 곧 단말기의 배터리 누수 문제를 초래하며 이것을 방지하기 위해서 Google Cloud Server가 대신해서 푸쉬 알람을 날리게 되는 것입니다.

Service

스레드와 비슷한 개념이지만 앱이 종료되면 같이 종료되는 스레드와 다르게 서비스는 독립적으로 작동될 수 있습니다. 즉, 독립적인 스레드라고 생각하면 됩니다.

App이 꺼져도 작동되는 푸쉬 알람, 파일 다운로드, 날씨와 같은 것을 Service라고 생각하면 됩니다. 또한 Service를 악용하게 된다면 개인 정보 유출 문제가 발생할 수 있어서 조심해서 사용해야 합니다.

Service Bind

서비스는 원래 독립적인 특성 즉, Main에 의해서 통제를 받지 않는 조그마한 프로세스라고 할 수 있습니다.
하지만, 서비스에 접근을 해서 데이터나 기능을 실행시킬 때 필요한 것이 Bind 입니다. 따로 IPC 통신이라고도 합니다.

  1. 독립적인 서비스
  • 위 그림의 왼쪽 부분이 독립적인 서비스이며, onCreate() -> onStartCommand() -> onDestroy() 의 총 3 단계로 이루어져 있습니다.
  • onStartCommand()가 존재합니다.
  1. Bind 서비스
  • 우측의 그림이 Bind 서비스이며, onCreate() -> onBind() -> onUnbind() -> onDestroy() 의 총 4 단계로 이루어져 있습니다.
  • onBind()와 onUnbind()가 존재합니다.

앱이 서비스에 어떻게 Bind 되는지 실습을 해보았습니다.
