클래스 및 인터페이스

코틀린의 클래스 및 인터페이스는 자바와 유사한 부분이 많지만, 새로운 개념이 추가되거나 약간 다른 방식으로 사용하는 부분도 있습니다. 이번에는 코틀린의 클래스 및 인터페이스를 자바와 비교하면서 공부해보겠습니다.

클래스와 인터페이스의 선언 및 인스턴스 생성

클래스와 인터페이스를 생성하는 방법은 자바와 거의 동일합니다.

코틀린에서 접근 제한자를 지정하지 않은 경우 이는 public으로 간주됩니다. 자바에서는 클래스 내부에 필드 혹은 메소드를 구현 여부와 무관하게 클래스 본체를 추가해야 합니다. 하지만, 코틀린은 선언된 내용이 없는 경우 클래스 이름만 선언할 수 있습니다. 또한, 인터페이스도 클래스와 동일한 규칙이 적용됩니다.

또한, 자바에서는 클래스의 인스턴스를 생성하기 위해 new라는 키워드를 사용했지만, 코틀린에서는 이를 사용하지 않습니다.

추상 클래스(abstract class)는 자바와 동일한 방법으로 선언하지만, 추상 클래스의 인스턴스를 생성하는 형태는 매우 다릅니다.

인터페이스를 선언하거나, 인터페이스의 인스턴스를 만드는 방법은 추상 클래스와 매우 유사합니다.

추상 클래스에서는 인스턴스 생성 시 생성자를 사용하지만, 생성자가 없는 인터페이스는 인터페이스 이름만 사용합니다.

프로퍼티

자바에서는 클래스 내에 자료를 저장하고 접근하기 위해 필드 (field) 와 메소드 (method) 를 사용합니다. 클래스에서 다루는 자료에는 이름과 주소 두 가지 항목밖에 없지만, 이 자료의 값에 접근하기 위해서 Getter/Setter 메소드를 추가해야 하므로 코드의 양은 불필요하게 증가하게 됩니다.

필드의 갯수가 적으면 상관 없지만, 필드의 수가 늘어날수록 이에 필요한 메소드가 배로 늘어나므로 코드 작성 시 불편함 또한 이에 비례하여 커집니다.

코틀린은 이러한 불편함을 개선하기 위하여 프로퍼티(property)를 사용합니다. 프로퍼티는 자료를 저장할 수 있는 필드와 이에 상응하는 Getter/Setter 메소드를 함께 제공하며, 자바의 필드와 유사한 형태로 선언합니다.

각 프로퍼티의 타입으로 String?을 사용했는데, 이는 해당 프로퍼티에 널이 들어갈 수 있음을 의미합니다.

코틀린의 프로퍼티 값도 값(val) 혹은 변수(var) 중 하나로 선언합니다. 위의 코드에서는 두 프로퍼티 모두 변수로 선언했는데, 이는 각 필드에 대한 Getter/Setter 메소드가 모두 존재하는, 즉 값이 언제든지 변할 수 있는 필드이기 때문입니다. 예를 들어, name 필드가 Getter 메소드만 있고 Setter는 없다면 값을 읽는 것만 가능합니다. 따라서 이는 val로 표현합니다.

코드를 보게 된다면 각 프로퍼티에 초기값으로 null이 명시적으로 할당된 것을 확인할 수 있습니다. 이처럼 코틀린에서는 클래스의 멤버 변수로 사용하는 프로퍼티는 초기값을 명시적으로 지정해야 하며, 그렇지 않을 경우 컴파일 에러가 발생합니다. 단, 생성자에서 프로퍼티의 값을 할당한다면 선언 시 값을 할당하지 않아도 됩니다.

프로퍼티의 선언 시점이나 생성자 호출 시점에 값을 할당할 수 없는 경우에는 lateinit키워드를 사용하여 이 프로퍼티의 값이 나중에 할당될 것임을 명시합니다. lateinit 키워드는 var 프로퍼티에만 사용 가능하며, 선언 시점에 값 할당을 요구하는 val 프로퍼티에는 사용할 수 없습니다.

lateinit 키워드를 사용항 프로퍼티를 초기화 없이 사용하려 한다면 Uninitialized PropertyAccessException 예외가 발생합니다. 이는 자바의 널 포인터 예외와 마찬가지로 컴파일 단계에서는 확인이 불가능합니다.

추가로, 프로퍼티에 초깃값을 할당하는 시점에서 해당 프로퍼티의 타입을 추론할 수 있다면, 타입 선언을 생략할 수 있습니다.

접근 제한자

자바에서 클래스 및 메소드, 필드의 가시성을 제한하기 위해 접근 제한자를 사용하는 것처럼, 코틀린에서도 접근 제한자를 사용하여 클래스와 함수, 프로퍼티의 가시성을 제어합니다.

제한자가 없으면 public으로 간주하므로 이를 생략하는 것을 권합니다. 자바에서는 접근 제한자를 생략하면 이에 대한 접근 범위를 패키지 단위로 제한합니다. 즉, 동일한 패키지 내에 있는 클래스에서만 접근이 가능합니다.
하지만, 이 제약은 그리 견고하지 않으며 해당 클래스가 포함된 모듈이 아닐지라도 패키지를 동일하게 맞추면 패키지 단위로 제한된 클래스, 필드 및 메소드에 접근할 수 있습니다.

이러한 단점을 보완하기 위해서 코틀린에서는 internal 접근 제한자를 제공합니다. internal 접근 제한자는 동일한 모듈 내에 있는 클래스들로의 접근을 제한합니다. 따라서 외부 모듈에서는 이 접근 제한자로 선언된 요소에 접근할 수 없습니다.
