ListView의 문제점

  1. ViewHolder Pattern을 강요하지 않는다.
  2. vertical 모드만 지원한다.
  3. Animation을 따로 지원하지 않기에 뷰에 대해 직접 구현해야 한다.
  4. 데이터가 변경될 때 notifyDataSetChanged()를 통해 뷰를 갱신하는데 모든 뷰를 갱신하기에 비용이 매우 크다.

위와 같은 문제점으로 인해 더 효율적인 리스트를 구현하기 위해서 RecyclerView가 등장하게 되었다. 더 자세한 내용은 아래의 글을 참고하자.

Context란

Context는 애플리케이션의 정보에 접근하기 위한 인터페이스이다. Context를 사용하면 getPackageName(), getResource 등과 같이 애플리케이션에 대해 시스템이 관리하는 정보에 접근할 수 있으며 startActivity(), bindService 등과 같이 안드로이드가 제공하는 시스템 서비스를 사용할 때도 사용된다.

보통 다른 프로그램에서 시스템 레벨에서 제공하는 전역 정보에 접근하기 위해 System 정적 클래스에 접근하는 것과 비슷한 맥락이다. 안드로이드에서는 애플리케이션 관리를 시스템에서 하는 것이 아니라 별도의 ActivityManagerService라는 또 다른 애플리케이션이 관리하고 있기 때문에 애플리케이션과 연관된 시스템 레벨의 함수를 호출하려면 ActivityManagerService를 통해서 해야 한다. 따라서 ActivityManagerService에게 자신이 어떤 애플리케이션인지 알리고 시스템 레벨의 함수에 접근하기 위해서 Context 객체가 필요한 것이다.

MVC,MVP,MVVM

여기서 설명하는 패턴은 아키텍처 패턴이라고 부른다.

  1. MVC

MVC는 Model과 View, Controller 구조를 갖는 디자인 패턴이다. Model에서 데이터와 상태, 비즈니스 로직을 구현하며 _View_와 _Controller_에 종속되지 않고 여러 곳에서 재사용될 수 있다.

View는 화면을 구현하는 컴포넌트이다. _Model_에 대한 이해없이 오로지 사용자의 인터렉션을 받아 반응한다. View와 _Model_은 최대한 종속되지 않고 독립적이게 구현해야 한다.

Controller는 _View_와 _Model_을 연결하는 역할을 한다. _View_에서 사용자와의 인터렉션이 발생하면 컨트롤러가 이것을 인지하고 모델을 갱신하는 역할을 한다.

안드로이드에서 _Controller_는 주로 액티비티에 해당하고 View는 XML 파일이 되며 Model은 데이터 클래스 등이 된다. 원래 MVC는 _View_와 _Model_을 완전히 분리시키고 _Model_이 어디에도 종속되지 않지만 안드로이드에서 _View_와 _Controller_가 액티비티에 같이 구성되어 있어 사실상 하나의 모습을 보인다. 따라서 _View_와 _Model_이 의존적이게 되어 테스트가 어렵고 액티비티 안에 대부분의 처리가 구현되어 스파게티? 코드가 될 수 있다.

  1. MVP

MVP는 _MVC_의 문제점을 보완하고자 나온 디자인 패턴이다. MVP에서는 액티비티를 _View_의 일부로 간주하여 액티비티 안에 대부분의 처리가 구현되는 것은 극복하였다. 액티비티가 View 인터페이스를 구현되어 있기에 가상 뷰를 만들어 _presenter_와 연결시키면 유닛 테스트가 가능하다.

presenter는 MVC의 _Controller_와 역할이 거의 같지만 _View_와 연결되어 있지 않고 그저 인터페이스라는 차이가 있다. _View_와 _Model_을 완전히 분리시켜줌으로써 서로의 독립성을 보장한다는 점이 장점이다. 하지만 _presenter_도 시간이 지남에 따라 비즈니스 로직이 모이는 경향을 보인다. 따라서 문제가 발생하기 쉽고 분리하기도 어려워 유지보수가 어려워진다는 단점이 있다.

  1. MVVM

MVVM은 DataBinding을 이용함으로써 모듈화를 쉽게하고 뷰와 모델을 연결하는 코드를 줄일 수 있는 장점을 가진 디자인 패턴이다. _View_가 _ViewModel_에 의해 바인딩되어 _ViewModel_의 값이 변경되면 _View_가 갱신된다. ViewModel은 Model을 Wrapping하는 구조이며 동시에 View가 Model에 이벤트를 전달할 수 있는 훅 기능을 한다.

Command Pattern과 _DataBinding_을 사용하여 View와 ViewModel의 의존성을 완전히 제거한다는 장점이 있다. MVVM 패턴 또한 시간이 지남에 따라 관계 없는 로직이 늘어나 xml 파일에 코드가 증가하는 문제가 발생한다. 이것은 유지보수를 힘들게 하기 때문에 뷰 바인딩 표현식에 값을 계산하지 않고 ViewModel에 직접 값을 가져와 처리해야 한다.

SparseArray와 HashMap 차이

Java에서의 Map은 편리하고 많이 사용된다.(보통 HashMap)
Android performance tuning 이야기를 할 때 HashMap 대신 SparseArray를 사용하면 도움이 된다고 한다. 그렇다면 무엇인지는 모르지만 들어보기만 했던 SparseArray를 보기에 앞서 Map의 개념부터 살펴보자.

  1. Map

Map의 핵심 개념은 get(key)와 put(key, value)이다.
Map은 인터페이스이기 때문에 구현은 몰라도 사용하는 입장에서는

위의 코드를 실행했을 때 value에 "강남스타일"이 들어있으면 Map이라고 생각할 수 있다. 그리고 가장 많이 사용하는 Map 구현체는 HashMap이다. 이는 key.hashCode()의 정수(integer) 결과값을 index % bucket_size 하여 bucket을 선택하게 된다.

하지만 대부분의 경우 hash collision이 발생할 가능성이 있다.

이는 무한개의 key를 hashCode() 함수를 통해 한정된 int 범위내의 값으로 표현하기 때문에 충돌은 필연적으로 발생할 수 밖에 없다. 즉, 같은 hashCode()를 사용한 경우에 "구하라"도 100, "함은정"도 100이 나올 수 있다.
이 경우 key는 분명 다르지만 같은 100번 bucket의 값을 반환하게 되는 문제가 발생한다.

이를 해결하기 위해 보통은 chaining이라는 기법을 사용하는데 bucket에 값을 직접 두지 않고 Linked list를 참조하게 한다.

위의 그림에서 "John Smith"와 "Sandra Dee"의 hash 값이 152로 충돌한다. 하지만 get("Sandra Dee")를 호출하면 내부적으로 152번 bucket을 통해 검색하려는 Key가 "John Smith"인지 확인하고(key.equals(“John Smith”)) 실패하니 다음 "Sandra Dee"가 맞는지 확인한다. 이제 맞으니 521-9655를 반환하게 된다.

Key의 성격에 따른 hash function만 잘 구현되어 있으면 매우 잘 동작한다. 하지만 반대로 key.hashCode() 구현을 적절히 구현하지 못한 경우 모두 같은 bucket을 선택하게 되고 결국 linked list와 별반 차이가 없어질 수 있다.

자, 그렇다면 여기까지 HashMap인데 그럼 SparseArray는 왜 더 빠르다는 것일까??

  1. SparseArray

일단, Sparse가 무엇을 의미하는지 알아야 한다. 이 단어는 말 그대로 데이터가 뜨문 뜨문하다라는 뜻이다. 예를 들어, Excel을 만든다 할 때 하나의 sheet에 최대 1048576 Row X 16384 Column을 가질 수 있는데 그럼 다음과 같이 2차원 배열로 구현할까??

Cell[][] cellMatrix = new Cell[1048576][16384];

물론 그렇지 않다. 위의 경우 실제 객체 생성을 제외해도 reference 하나당 4byte이니 68기가 정도를 할당해야한다. 보통은 왼쪽 위에 자료가 몰려 있긴 하지만 왼쪽위와 오른쪽 아래에 찔끔씩 데이터가 존재해도 효율적으로 처리해야 한다. 이런 상황을 Sparse하다고 한다.

sparse matrix가 2차원 이야기라면 sparse array는 1차원에서의 이야기이다. integer를 index로 하는 array가 있을 때 0~100번 index에 데이터가 또, 200~250 사이에도 데이터가 있을 수 있다. (비어있는 공간이 적지만 커지면 어마어마하다.)나머지는 다 비어있다면 자료 구조를 어떻게 설계할 것인가?

이에 대한 답은 여러가지가 있을 수 있겠지만 Android에서는 java.android.util.SparseArray이라는 구현체를 제공한다. Code도 Apache license로 공개되어 있으며 간단한 편이다.

당연히 SparseArray가 만능은 아닌데 HashMap에 비해 가장 큰 제약은 Key가 정수여야만 한다는 점이다. 더 정확히는 int형일 때 SparseArray를 사용하고 long일 때는 LongSparseArray를 사용한다. 이로 인해 hash function을 수행할 필요가 없고 hash를 사용하고 있지 않으니 hash collision이 없으니 list iteration과 equals()를 수행할 필요가 없다. 뿐만 아니라 int나 long인 primitive key는 즉, 기본 자료형을 Key로 사용함으로써 Wrapper 클래스로 변환하는 boxing/unboxing의 오버헤드도 제거된다.

SparseArray는 내부적으로 작은 배열 두개(key를 저장하는 int[], value를 저장하는 object[])를 사용하여 값을 저장한다.(HashMap은 Hash Bucket이라는 큰 해시 배열을 통해 값을 저장한다.)

성능 향상을 위해서 SparseArray는 데이터가 지워지면, deleted로 표시하고 그 공간을 비워놓는다. 물론, 지워진 데이터의 키값을 주고 데이터를 추가한다면 그 자리는 다시 채워지게 된다. 이러한 방식이 비워진 데이터를 다시 정렬하는 과정에서의 충돌이 생기는 HashMap 방식보다 메모리를 효율적으로 사용해서 성능이 더 좋다.

일반 배열처럼 인덱스로 접근하여 키와 벨류를 가져올 수 있다. 하지만 단점도 존재한다. 내부적으로 데이터가 뜨문뜨문하게 저장되기 때문에 탐색시 이진 탐색을 톨하여 키를 찾는다. 그렇기에 데이터가 많은 구조에 사용하기 부적합하며 배열의 항목을 삽입/삭제해야 하기 때문에 HashMap보다 속도가 느리다. 최대 수백개까지의 항목을 보유한 컨테이너일 때 사용하기를 권장한다.
