ENUM 사용을 자제시키는 이유

ENUM의 각 값은 객체이며 각 선언은 단순히 객체를 참조하기 위해 런타임 메모리를 사용한다. 따라서 정수 또는 문자열 상수보다 더 많은 메모리를 차지하게 된다. 게다가 단일 ENUM을 추가하면 최종 DEX 파일 크기를 증가시키기에 런타임시 오버헤드가 발생할 수 있고, 앱의 크기가 증가하게 된다. 안드로이드에서는 ENUM 대신 TypeDef 어노테이션을 사용한다.

대용량 Bitmap을 불러올 경우 메모리 문제를 해결하는 방법

createScaledBitmap(Bitmap bitmap, int width, int height)
비트맵을 생성할 때 작은 크기로 생성하여 메모리 사용을 줄일 수 있다. 허나 이미 원본 비트맵이 메모리에 로드되어 있어야 리사이즈된 비트맵을 생성할 수 있다는 단점이 있다.

  1. Bitmap.Options.inSampleSize

inSampleSize는 2의 지수 값만 가질 수 있으며, 2의 지수만큼 이미지를 작게 만든다. inSampleSize 크기만큼 픽셀을 건너뛰어 리사이징하기 때문에 속도가 매우 빠르다. 허나 2의 지수가 아닌 값으로는 리사이징을 못한다는 단점이 있다.

  1. Bitmap.Options.inScaled / Bitmap.Options.inDensiy

어떠한 사이즈로든 리사이징 되고 리사이징 필터가 적용되어 더욱 정교한 리사이징이 가능하다. 하지만 추가적인 필터단계는 많은 시간 소요가 발생하기 때문에 inSampleSize 방법에 비해 느리다.

  1. Combine inSampleSize & inScaled & inDensity

원하는 이미지 크기보다 2배 큰 이미지를 inSampleSize를 통해서 리사이징한다. (2의 지수만큼 리사이징 가능하므로) 원하는 크기까지 inScaled와 inDensity를 이용하여 정교하게 리사이징하여 원하는 크기의 이미지를 얻는다.

  1. Bitmap.Options.inJustDecodeBounds

원본 Bitmap 객체를 생성하지 않은 채로 원본 이미지 크기를 구할시 inJustDecodeBounds 옵션을 이용한다. 이것의 값이 true일 경우 BitmapFactory.decodeFile(fileName, Options)를 통해 Bitmap을 생성시 Bitmap 객체를 반환하지 않고 Bitmap 정보를 Options 객체에 담는다.

따라서 Options.outWidth, Options.outHeight를 통해 너비와 높이를 알 수 있다. 반대로 Bitmap 객체를 생성하고 싶을 경우 inJustDecodeBounds 값을 false로 설정하여 decode하면 객체를 반환한다.

프로세스와 스레드 차이

프로그램이 메모리에 올라가 실행되면 프로세스라고 부른다. 프로세스는 운영체제로부터 메모리와 CPU를 할당받게 된다. 쓰레드는 프로세스 내에서 동작하는 실행흐름을 말한다. 따라서 쓰레드끼리 프로세스 내의 스택을 제외한 자원을 공유할 수 있다. 기본적으로 하나의 프로세스가 실행되면 하나 쓰레드가 실행되며, 이것을 메인 쓰레드라고 부르며, 안드로이드에서는 UI 쓰레드라고 부른다.

여러 프로세스를 통해 작업을 하지 않고 쓰레드를 사용하는 이유는?

첫 번째, 프로세스 간 자원을 공유하지 않는다. 프로세스는 자신만의 메모리 영역이 존재하고 이것은 다른 프로세스에서는 접근이 불가능하다.

두 번째, 프로세스의 생성은 많은 자원과 메모리가 사용되므로 비용이 크다. 프로세스가 실행되려면 메모리에 올라와야 하므로 비용이 크다.

세 번째, 다른 프로세스에게 CPU를 넘기는 Context Switching은 비용이 큰 작업이고, 시스템 콜을 발생시키므로 시간이 오래 걸린다.

따라서 여러 작업을 동시에 실행하기 위해서는 여러 쓰레드를 사용하여 비동기적인 실행을 하는 것이 더욱 효율적인 방법이다.
